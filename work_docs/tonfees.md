## Почему Jetton дороже нативного TON

Jetton — это не нативный transfer. Каждая "передача" — это цепочка сообщений между контрактами:

```
кошелёк → Jetton wallet отправителя → Jetton wallet получателя → (notify)
```

Нативный TON transfer — **одно сообщение**. Jetton transfer — **минимум три**. Каждый hop платит за вычисления и forward. При N получателях в Splitogram цепочка разветвляется, и расход растёт линейно.

Эмпирическая формула из текущих данных:

```
attach ≈ 0.05 TON (base) + 0.09 TON × N (получателей)
```

## Как снизить газ

### 1. Убрать forward_amount там, где он не нужен

Если получатели — обычные кошельки (не контракты), то `forward_amount = 0` и никакого Jetton Notify не нужно. Это убирает один hop на каждого получателя — экономия ~0.02–0.03 TON на человека.

### 2. Снизить стартовый attach

Сейчас attach = 0.5 TON — это избыток. Достаточно 0.35–0.40 TON для двух получателей. Excess всё равно вернётся, но меньше TON будет временно заморожено.

### 3. Масштаб — главный враг

| Получателей | Примерный газ |
| ----------- | ------------- |
| 2           | ~0.09 TON     |
| 3           | ~0.18 TON     |
| 5           | ~0.36 TON     |

При групповых расчётах на 5+ человек стоит задуматься об архитектуре: например, отправлять один transfer на промежуточный контракт, а он уже раздаёт — но это усложняет логику и добавляет риски.

## Как устроены комиссии в TON

### Из чего состоит газ

- **`gas_price`** — устанавливается валидаторами через on-chain конфигурацию (Config 21 для masterchain, Config 25 для basechain). Может меняться голосованием валидаторов.
- **Storage fee** — отдельная плата за хранение данных в контракте, тарифицируется за каждую секунду существования.
- **Forward fee** — плата за передачу сообщений между контрактами.

### Стабильны ли комиссии

Исторически — да, достаточно стабильны. Но это не гарантия: при росте нагрузки на сеть валидаторы могут пересмотреть `gas_price`. Если курс TON растёт, комиссия в USD может оставаться похожей даже при росте в TON.

## Как считать газ заранее

### Эмпирический подход (текущий)

Смотришь на реальные транзакции, считаешь excess, выводишь формулу. Работает, но хрупко — при изменении конфига сети формула поплывёт.

### Через estimate API (рекомендуется для продакшена)

TON SDK и TONAPI умеют делать dry-run транзакций — возвращают точный газ без реального исполнения:

- `GET https://tonapi.io/v2/blockchain/messages/estimate`
- `POST https://toncenter.com/api/v2/estimateFee`

**Рекомендуемая стратегия:**

1. Перед отправкой делаешь estimate через API.
2. `attach = estimate + 20% буфер` (на случай изменения конфига между estimate и исполнением).
3. `response_destination` всегда указывает на кошелёк инициатора — excess возвращается автоматически, переплата не теряется.

## Что реализовано

### Gas estimation через TONAPI emulate

**Файл:** `backend/src/services/tonapi.ts` → `estimateSettlementGas()`

Перед каждой транзакцией бэкенд:

1. Получает seqno кошелька через `GET /v2/blockchain/accounts/{address}/methods/seqno`
2. Определяет тип кошелька (V4R2 / V5R1) из `interfaces` в ответе TONAPI
3. Получает `wallet_id` (subwallet_id) для V5 кошельков через `GET /v2/blockchain/accounts/{address}/methods/get_subwallet_id`
4. Собирает внешнее сообщение (external message) с нулевой подписью:
   - V4R2: `signature(512 zero) + subwallet_id + valid_until + seqno + op + send_mode + ref(internal_msg)`
   - V5R1: `prefix(0x7369676E) + wallet_id + valid_until + seqno + Maybe(^C5_OutList) + has_extended(0) + signature(512 zero)` — W5 формат: подпись в конце (get_last_bits), действия как стандартный C5 OutList в Maybe ref
5. Отправляет в `POST /v2/traces/emulate?ignore_signature_check=true`
6. Рекурсивно суммирует `total_fees` по всем 11 транзакциям в trace

**Формула:**

```
gasAttach = FORWARD_TON + emulated_fees + 15% буфер     (если emulate сработал)
          = FORWARD_TON + EMPIRICAL_FEES + 25% буфер    (fallback)

FORWARD_TON     = 300_000_000  (0.3 TON)  — контракту нужно для 2 исходящих Jetton-трансферов
EMPIRICAL_FEES  = 100_000_000  (0.1 TON)  — консервативный fallback (emulate даёт ~0.035 TON)
```

- Emulate даёт точный газ на момент запроса — автоматически учитывает изменения `gas_price` валидаторами.
- Если emulate не работает (неизвестный тип кошелька, TONAPI недоступен) — используется эмпирический fallback.
- Буфер ниже для emulate (15%) чем для fallback (25%), потому что emulate точнее.
- `response_destination` = кошелёк инициатора → excess возвращается всегда.

## Когда и как менять константы

### Симптом: транзакции перестали проходить (out of gas)

**Если emulate работает** — газ считается на лету, и проблема не должна возникать. Emulate автоматически учитывает изменения `gas_price` сети. Если всё равно не хватает — увеличить `CONTINGENCY_PCT` в `settlements.ts` (сейчас 0.15 для emulate).

**Если emulate не работает** (возвращает null → используется эмпирический fallback):

1. Открыть упавшую транзакцию на [Tonviewer](https://tonviewer.com) (или testnet.tonviewer.com)
2. Посмотреть сколько TON было attached и сколько вернулось как excess
3. Фактический gas burn = attached − excess
4. Если burn > текущий `EMPIRICAL_FEES` (0.1 TON) — причина найдена

**Исправление fallback'а:**

- В `backend/src/api/settlements.ts` увеличить `EMPIRICAL_FEES` до нового значения (округлённого вверх)
- 25% буфер пересчитается автоматически
- Задеплоить бэкенд

**Диагностика emulate:** проверить логи бэкенда — если `estimateSettlementGas` всегда возвращает null:

- Тип кошелька не V4/V5 → `walletInterfaces` в ответе TONAPI пуст или содержит другую версию
- TONAPI не доступен или rate-limited → проверить `TONAPI_KEY`
- Формат внешнего сообщения устарел → сверить с текущей спекой W5 (`tonkeeper/w5` repo)

### Когда газ может вырасти

| Причина                                        | Обрабатывается emulate?   | Действие если нет                                        |
| ---------------------------------------------- | ------------------------- | -------------------------------------------------------- |
| Валидаторы повысили `gas_price` (Config 21/25) | Да, автоматически         | Увеличить `EMPIRICAL_FEES` (только для fallback)         |
| Рост forward_fee из-за нагрузки на сеть        | Да, автоматически         | Увеличить `EMPIRICAL_FEES` (только для fallback)         |
| Изменение структуры контракта (новая версия)   | Да, если контракт деплоен | Перезамерить на тестнете, обновить `EMPIRICAL_FEES`      |
| Новая версия кошелька (V6+)                    | Нет                       | Добавить `buildV6Body()` в `tonapi.ts`                   |
| Добавление третьего получателя в контракт      | Да, автоматически         | `FORWARD_TON` += 150_000_000 (ещё 0.15 на доп. трансфер) |

### Как перезамерить (для fallback)

```bash
# 1. Сделать тестовый settlement на testnet
# 2. Найти транзакцию на tonviewer
# 3. Посмотреть цепочку сообщений, посчитать:
#    gas_burn = attached - sum(excess_returned)
# 4. Обновить EMPIRICAL_FEES = ceil(gas_burn / 10_000_000) * 10_000_000
#    (округление вверх до 0.01 TON)
```

### Безопасность переплаты

Переплата газа **безопасна** — excess всегда возвращается на кошелёк инициатора через `response_destination`. Пользователь видит корректную сумму перед подтверждением. Лучше переплатить и вернуть, чем недоплатить и сломать транзакцию.
